# M6 ‚Äî Engagement & Reaction System

> **Goal**: By the end of M6, every post, every individual list item, and every comment supports the üî• Fire reaction ‚Äî a per-user toggle that feels instant via optimistic UI. The Verdict Meter system is fully operational, calculating and displaying a community verdict (`CONFIRMED` / `CONTESTED` / `HOT TAKE` / `DEBUNKED` / `UNDECIDED`) on every approved post. Per-item challenges (‚ö°) allow any user to open a targeted debate thread on a specific ranked item. Any post that has been countered shows a "Rivals (N)" button, leading to a full Battle View page where both the original and rival lists sit side-by-side. Reaction counts are denormalized and updated atomically in the database on every toggle.

**Key architectural decisions:**
- Reactions use a **toggle model** with a DB-level `UNIQUE (user_id, target_type, target_id, reaction_type)` constraint. Calling the endpoint twice toggles the reaction off ‚Äî only one row per user per target ever exists.
- `fire_count` is a **denormalized counter** on `posts`, `list_items`, and `comments` ‚Äî an integer column updated atomically via SQL `SET fire_count = fire_count + 1`. Never recomputed from a `COUNT()` scan on every read.
- Verdict recalculation is **synchronous for MVP** ‚Äî triggered inline after every successful reaction write or challenge open. At scale (later milestone) this moves to a background Redis queue.
- Per-item challenges reuse the `comments` table with `list_item_id` set. No separate table needed ‚Äî the storage layer is already built.
- The `challenge_count` on both `posts` and `list_items` is incremented only for **root-level** item-anchored comments (depth = 0). Replies to challenges do not count.
- All count mutations use **optimistic UI** on the frontend ‚Äî the number updates in the browser before the API call completes, then rolls back silently on failure.

## Step 1 ‚Äî üíª LOCAL: Design the Verdict Calculation Engine

**`backend/app/services/verdict.py`**

The Verdict Meter is the platform's signature feature. It is NOT just a vote count ‚Äî it is a weighted engagement signal.

### Input signals:
| Signal | Meaning |
|---|---|
| `post.fire_count` | Raw community approval |
| `post.challenge_count` | Number of ‚ö° challenges opened on this post's items |
| `post.comment_count` | Total comment volume |
| `post.rival_count` | Number of Counter Lists published against this post |
| `sum(list_item.fire_count)` | Item-level distributed approval |

### Verdict Decision Matrix:

| Condition | Verdict |
|---|---|
| Total engagement below minimum threshold (`fire < 10`, `challenge < 2`, `rival < 1`) | `UNDECIDED` |
| High fire, low challenge ‚Äî `fire ‚â• 20` AND `fire > challenges √ó 5` | `CONFIRMED` |
| Very high challenge relative to fire ‚Äî `challenges > fire / 3` AND `rival_count > 0` | `DEBUNKED` |
| Both high ‚Äî `fire ‚â• 15` AND `challenges ‚â• 5` | `CONTESTED` |
| Extreme challenge volume ‚Äî `challenges ‚â• 10` regardless of fire | `HOT_TAKE` |
| Admin manually overrides verdict from admin panel | (any, persists) |

All thresholds live in a `VERDICT_THRESHOLDS` constant dict in `app/core/config.py` so they can be adjusted at runtime by changing the env or config without touching logic code.

### Function signature:
```
def recalculate_verdict(db: Session, post_id: UUID) -> VerdictStatus:
    - Fetches post + all list_items in one query
    - Sums item-level fire counts
    - Applies threshold matrix
    - Writes new verdict back to post.verdict
    - Commits
    - Returns the new VerdictStatus
```

This function is called at the tail of:
- `POST /reactions` (if target is a post or list item)
- `POST /posts/{id}/items/{item_id}/challenge` (new challenge opened)
- `POST /comments` (any new comment ‚Äî comment_count impacts verdict, M7 calls this)

> **Why synchronous?** At launch volumes (hundreds of posts), synchronous recalculation adds ~2ms to the response time. Acceptable. At thousands of concurrent fires per second, this moves to an async worker. Cross that bridge later with real data.

## Step 3 ‚Äî üíª LOCAL: Build the Reaction CRUD Layer

**`backend/app/crud/reaction.py`**

| Function | What it does |
|---|---|
| `get_reaction(db, user_id, target_type, target_id)` | Return the `Reaction` row if it exists, else None |
| `create_reaction(db, user_id, target_type, target_id, reaction_type)` | Insert a new Reaction row |
| `delete_reaction(db, reaction)` | Delete the Reaction row |
| `increment_fire_count(db, target_type, target_id)` | Atomic SQL increment on the correct table |
| `decrement_fire_count(db, target_type, target_id)` | Atomic SQL decrement (floor at 0) |
| `get_fire_count(db, target_type, target_id)` | Fetch current fire_count from the target row |
| `get_batch_reaction_state(db, user_id, target_type, target_ids)` | Return dict of {target_id: bool} for bulk-checking |

### Atomic increment pattern (critical ‚Äî prevents race conditions):
```python
db.query(Post).filter(Post.id == target_id).update(
    {"fire_count": Post.fire_count + 1},
    synchronize_session=False
)
db.commit()
```
This generates `UPDATE posts SET fire_count = fire_count + 1 WHERE id = ?` at the SQL level ‚Äî atomic, no race condition. Never read the count, add 1, then write.

### Batch reaction state ‚Äî prevents N+1 on feed load:
Instead of one `GET /reactions/{id}` per visible post, the frontend sends all visible post IDs at once:
```python
# Single query: SELECT target_id FROM reactions WHERE user_id = ? AND target_type = ? AND target_id IN (...)
rows = db.query(Reaction.target_id).filter(
    Reaction.user_id == user_id,
    Reaction.target_type == target_type,
    Reaction.target_id.in_(target_ids),
).all()
reacted_ids = {str(row.target_id) for row in rows}
return {str(tid): (str(tid) in reacted_ids) for tid in target_ids}
```

## Step 4 ‚Äî üíª LOCAL: Build the Reaction Service

**`backend/app/services/reaction.py`**

Single orchestration function that wraps the full toggle flow:

```
def toggle_reaction(db, user_id, target_type, target_id, reaction_type) ‚Üí ReactionResponse:

    existing = crud.get_reaction(db, user_id, target_type, target_id)

    if existing:
        crud.delete_reaction(db, existing)
        crud.decrement_fire_count(db, target_type, target_id)
        reacted = False
    else:
        crud.create_reaction(db, user_id, target_type, target_id, reaction_type)
        crud.increment_fire_count(db, target_type, target_id)
        reacted = True

    new_count = crud.get_fire_count(db, target_type, target_id)

    # Recalculate verdict only for post-level or item-level reactions
    verdict = None
    if target_type in (ReactionTargetType.post, ReactionTargetType.list_item):
        post_id = resolve_post_id(db, target_type, target_id)  # resolve item ‚Üí post
        new_verdict = verdict_service.recalculate_verdict(db, post_id)
        verdict = new_verdict.value

    return ReactionResponse(reacted=reacted, fire_count=new_count, verdict=verdict, ...)
```

The helper `resolve_post_id` looks up `ListItem.post_id` if target is a list_item, or returns the ID directly if it's a post.

## Step 5 ‚Äî üíª LOCAL: Build the Reaction API Endpoints

**`backend/app/api/v1/reactions.py`**

### POST /reactions ‚Äî Toggle a reaction
- **Auth required**
- Validates that `target_id` actually references a real row (prevents reactions on ghost IDs)
- Calls `reaction_service.toggle_reaction()`
- Returns `ReactionResponse` ‚Äî always HTTP 200 regardless of outcome (the `reacted` field communicates the result)
- Anonymous users (who still have JWT sessions from M3) can react ‚Äî no email required

### GET /reactions/state ‚Äî Batch reaction state
- **Auth required**
- Query params: `target_type` (string), `target_ids` (comma-separated UUIDs, max 100)
- Returns `{ "states": { "uuid1": true, "uuid2": false, ... } }`
- Called once when the feed mounts with all visible post IDs

### GET /posts/{id}/verdict ‚Äî Get current verdict
- **No auth required**
- Returns `{ "verdict": "contested", "fire_count": 48, "challenge_count": 12, "rival_count": 3 }`
- Used to refresh the verdict meter without refetching the entire post

Wire reactions router under `/reactions` prefix in `api/v1/__init__.py`.

## Step 6 ‚Äî üíª LOCAL: Build the Per-Item Challenge Endpoint

Per-item challenges are stored in the `comments` table as item-anchored root-level comments. There is no separate "challenge" router ‚Äî this logic lives inside the **Comments service** (M7), but we expose a dedicated convenience endpoint here for clarity and discoverability.

**`POST /posts/{id}/items/{item_id}/challenge`** (auth required)
- Validates post and item exist
- Creates a `Comment` row with `list_item_id = item_id`, `post_id = post_id`, `depth = 0`, `parent_comment_id = None`
- Increments `list_item.challenge_count` atomically
- Increments `post.challenge_count` atomically
- Increments `post.comment_count` atomically
- Triggers `verdict_service.recalculate_verdict()` ‚Äî a challenge heavily influences the verdict
- Returns the created Comment as `CommentResponse`

> Note: Only root-level item challenges (depth = 0, no parent) increment `challenge_count`. A reply to a challenge is just a regular reply ‚Äî it increments `comment_count` but not `challenge_count`.

**`GET /posts/{id}/items/{item_id}/challenges`** (no auth required)
- Returns `List[CommentResponse]` filtered by `list_item_id = item_id` and `parent_comment_id = null`
- These are the root-level challenge threads anchored to a specific item
- Each challenge can then have nested replies via the standard comment reply endpoint (M7)

### Optimistic update implementation:
1. User clicks üî•
2. Immediately flip `isReacted` boolean and `¬±1` the displayed count in local state
3. Fire `POST /api/reactions` in background
4. On **success**: do nothing ‚Äî state already correct
5. On **error**: revert count and `isReacted`, briefly apply a red shake CSS class to signal failure
6. If `onVerdictUpdate` callback is provided and the response includes an updated `verdict`, call it to update the parent post's Verdict Meter

### Hydrating state on feed load:
On the feed `useEffect`, call:
```typescript

```
Store the resulting map in a Zustand store: `{ [postId]: boolean }`.
Each `FireButton` reads its initial `isReacted` from this store on mount ‚Äî no individual per-post fetch.
