# M5 — Post Submission, Feed System & Single Post View

> **Goal**: By the end of M5, authenticated users can submit all 8 post types through a rich multi-step composer, those posts enter the review queue, and the public feed shows approved posts newest-first with full pagination. Any visitor (logged in or not) can open a single post page and read the full ranked list with all item justifications, sources, and images. Counter Lists publish instantly without review. The 2-hour edit window begins the moment a post is approved.

---

## Architecture Overview

```
Browser
│
├── /submit                  → Multi-step Post Composer (auth required)
│   ├── Step 1: Choose post type
│   ├── Step 2: Fill in title + intro + category + visibility
│   ├── Step 3: Add list items (rank, title, description, image per item)
│   ├── Step 4: Add sources (label + URL, optional but badge-rewarded)
│   └── Step 5: Preview + Submit
│
├── /                        → Public Feed (paginated, any visitor)
│   ├── Approved posts, newest first
│   ├── Filter by category
│   └── Author posts float higher
│
├── /private                 → Connection Feed (auth required)
│   └── Posts from followed/connected users (any status visible to you)
│
└── /post/[id]               → Single Post Page (SSR)
    ├── Full list with ranked items, justifications, images
    ├── Source citations
    ├── Verdict Meter placeholder (filled in M6)
    ├── Counter List references (if any)
    └── Edit button visible to author within edit window

FastAPI Backend
│
├── POST /posts                    → Submit a new post (auth required)
├── GET  /posts                    → Paginated public feed (public)
├── GET  /posts/feed               → Connection feed (auth required)
├── GET  /posts/{id}               → Single post detail (public, privacy-gated)
├── PATCH /posts/{id}              → Edit a post (auth, within edit window)
├── DELETE /posts/{id}             → Delete own post (auth, soft-delete)
└── GET  /posts/{id}/counter-lists → Fetch counter/rival lists for a post
```

**Key architectural decisions:**
- All post types share the **same `Post` table** — type is encoded in the `post_type` enum column. This avoids denormalization.
- List items (`list_items` table) are always ordered by `rank`. For `this_vs_that` and `who_is_better`, items are still list items — just with 2–3 items and differing display semantics.
- Counter Lists use `counter_to_post_id` FK to reference the parent — they skip the review queue.
- The `edit_deadline` field is set at approval time: `approved_at + 2 hours`. After that, the backend returns 403 on any PATCH attempt.
- Image upload for list items flows through the same MinIO pipeline used in M4 for avatars — resized to 1280×720 (wide format), stored under `posts/{post_id}/items/{item_id}/{ts}.jpg`.

## Step 1 — 💻 LOCAL: Build the Post Schemas

**`backend/app/schemas/post.py`** — four schema classes:

### PostItemRequest (embedded inside PostCreateRequest)
What each list item looks like when submitted:
```
rank: int                  (1-based, enforced sequential by validator)
title: str                 (max 300 chars)
description: str           (the justification — max 2000 chars, markdown allowed)
image_url: str | None      (CDN URL, uploaded separately via POST /posts/items/image)
source_label: str | None   (per-item optional source)
source_url: str | None
```

> Items with empty `description` must be rejected by backend validation. Every item must justify itself.

### PostCreateRequest
The full payload sent when a user submits a post:
```
title: str                 (max 300 chars)
post_type: PostType        (one of the 8 enum values)
intro: str | None          (max 2000 chars — the list's intro paragraph)
category_id: UUID | None   (must reference a valid categories row)
visibility: PostVisibility (public / connections_only / private)
counter_to_post_id: UUID | None   (if this is a Counter List, reference the original)
items: List[PostItemRequest]      (min 1, max 25 — validated in backend)
sources: List[PostSourceRequest]  (optional, globally attached sources)
```

> **Validation rules** the backend must enforce:
> - `top_list`, `best_of`, `worst_of`, `hidden_gems`: items must be between 3 and 25
> - `this_vs_that`: exactly 2 items required
> - `who_is_better`: 2–5 items required
> - `fact_drop`: 1–10 items (each item = one fact)
> - `counter_list`: `counter_to_post_id` must be provided and resolve to a real approved post

### PostResponse
What the API returns when fetching a post:
```
id: UUID
title: str
post_type: str
status: str
visibility: str
verdict: str
intro: str | None
author: UserProfile (embedded — see M4)
category: CategoryInfo | None
items: List[PostItemResponse]
sources: List[PostSourceResponse]
fire_count: int
comment_count: int
challenge_count: int
rival_count: int
edit_deadline: datetime | None     (only visible to post author)
is_editable: bool                  (true if author + within edit window)
created_at: datetime
approved_at: datetime | None
counter_to_post: PostSummary | None  (lightweight embed of original post, if counter list)
counter_lists: List[PostSummary]     (lightweight list of any rivals)
```

### PostFeedItem (for feed endpoints — lighter than full PostResponse)
Strips down to what's needed for feed cards to keep payloads fast:
```
id, title, post_type, status, verdict, fire_count, comment_count,
challenge_count, is_sourced, category, author (name + avatar only),
created_at, approved_at, first_item_title (for preview text card)
```

## Step 2 — 💻 LOCAL: Build the Post CRUD Layer

**`backend/app/crud/post.py`** — the database access layer.

| Function | Purpose |
|---|---|
| `create_post(db, author_id, data)` | Insert post + items + sources in a single transaction |
| `get_post_by_id(db, post_id, viewer)` | Fetch post + related data; enforce visibility rules |
| `get_public_feed(db, category_id, cursor, limit)` | Approved posts, newest first, keyset-paginated |
| `get_connection_feed(db, viewer_id, cursor, limit)` | Posts from followed/connected users, keyset-paginated |
| `update_post(db, post, data)` | Partial update, enforces edit window check |
| `delete_post(db, post)` | Soft-delete: set status to 'hidden' |
| `get_counter_lists(db, post_id)` | Fetch all counter lists pointing to a post |
| `increment_field(db, model, id, field)` | Atomic increment for fire_count, comment_count, challenge_count |

### About keyset-based pagination
Do **not** use `OFFSET` pagination — it becomes unstable while new content is being inserted during browsing. Instead, use cursor-based pagination:
- Client sends `cursor` = last seen post's `created_at` as an ISO timestamp
- Backend does: `WHERE created_at < cursor ORDER BY created_at DESC LIMIT N`
- Response includes a `next_cursor` = `created_at` of the last item returned
- If `next_cursor` is null, the client knows there are no more pages

> This is the same approach used by Twitter, Instagram, and Reddit's modern API. It never shows duplicate content even as new posts arrive.

### create_post transaction pattern
All three tables must succeed or all three roll back:
1. Insert the `Post` row
2. For each item in `data.items`: insert a `ListItem` row with `post_id + rank`
3. For each source in `data.sources`: insert a `PostSource` row with `post_id`
4. If post is a `counter_list`: set `post.status = approved` immediately (bypass queue) and increment `counter_to_post.rival_count`
5. Commit — if any step raises, rollback all

## Step 3 — 💻 LOCAL: Build the Item Image Upload Endpoint

Before submitting a post, users upload item images one at a time. This is a separate endpoint so the composer can show a preview immediately without uploading the whole post.

**`POST /posts/items/image`** (auth required)

Flow:
1. User selects an image in the composer for item #3
2. Frontend calls `POST /posts/items/image` with the image as multipart/form-data
3. Backend runs the same validation + Pillow pipeline as avatar upload (M4 service layer)
4. Image is resized to **1280×720** (16:9 landscape ratio — better for list cards)
5. Stored in MinIO at `posts/temp/{user_id}/{ts}.jpg` (temp until post is actually saved)
6. Backend returns `{ "image_url": "https://cdn.yotop10.com/..." }`
7. Composer stores that URL in the item state — it's sent as part of `PostCreateRequest`

> Items that end up never submitted (e.g. user closes the composer) leave orphan images in MinIO temp storage. A scheduled cleanup job (M-later) sweeps `posts/temp/` for images older than 24 hours.

## Step 4 — 💻 LOCAL: Build the Post API Endpoints

**`backend/app/api/v1/posts.py`**

### POST /posts — Submit a new post
- Auth required
- Validate all items, sources, count rules
- Call `crud.post.create_post()`
- If Counter List → status set to `approved` immediately, `rival_count` on original incremented
- Otherwise → status = `pending_review`
- Return `PostResponse`

### GET /posts — Public feed
- No auth required
- Query params: `category` (UUID, optional), `cursor` (datetime string, optional), `limit` (1–50, default 20)
- Returns only `approved` posts visible as `public`
- Author posts: Posts by users with `is_author = True` receive a small score boost in the ordering. Implement as: first return approved Author posts in the last 2h, then all others. Keep it simple — no complex scoring.
- Include `next_cursor` in response envelope

### GET /posts/feed — Connection feed
- Auth required
- Shows posts from users the viewer follows or is connected with
- Includes all visibility levels (public, connections_only) — but NOT private posts from others
- Ordered by `created_at DESC` with same keyset pagination

### GET /posts/{id} — Single post detail
- No auth required
- Returns full `PostResponse`
- Applies privacy gate: if post visibility is `connections_only`, check if viewer is connected. If `private`, only the author can see.
- If post not found OR privacy blocks access → return 404 (never reveal the existence of private posts to strangers)

### PATCH /posts/{id} — Edit a post
- Auth required, must be post owner
- Checks `edit_deadline`: if `datetime.utcnow() > post.edit_deadline` → 403 with message "Edit window has closed."
- Allows updating: `title`, `intro`, `items` (content only — rank and item count immutable after approval)
- Does NOT allow changing `post_type`, `visibility`, `category`, or `counter_to_post_id`.

### DELETE /posts/{id} — Delete own post
- Auth required, must be owner OR admin
- Soft-delete: `status = 'hidden'` — do not hard-delete, items stay in DB for moderation audit trail

### GET /posts/{id}/counter-lists — Get rival lists
- No auth required
- Returns list of `PostSummary` objects (counter lists linked to this post)

## Step 5 — 💻 LOCAL: Build the Post Composer (Frontend)

**`frontend/src/app/submit/page.tsx`** — the multi-step composer.

This is the most complex frontend form in the application. Design it as a **stepped wizard** with progress indicators.

### Step 1: Post Type Selector
Show 8 large clickable cards, one per post type:
- Icon + Name ("Top List", "This vs That", etc.)
- Short description of what each type means
- When selected, it must visually highlight that card and enable the "Continue" button
- Post type is immutable once selected (user must start over to change it)

> UX note: A user who just wants to write "Top 10 Best Albums" should land on this screen and click "Top List" in under 5 seconds. No confusion. Make the type names clear and examples obvious.

### Step 2: Meta Information
Form fields:
- **Title** (max 300 chars, char counter shown)
- **Intro** (textarea, max 2000 chars, char counter — optional but strongly encouraged with a tooltip saying "A good intro earns more engagement")
- **Category** (dropdown of all categories from the DB)
- **Visibility** (radio: Public / Connections Only / Private)
- If post type is `counter_list`: show a search field to find and link the original post

### Step 3: List Items
This is a **drag-and-reorder list editor**. Each row represents one item.

Fields per item:
- Rank number (auto-generated and displayed, reordered by dragging)
- Title (max 300 chars)
- Description (rich textarea, max 2000 chars — the justification)
- Image Upload button (calls `POST /posts/items/image`, shows preview inline on success)
- Optional Source label + URL fields per item

Controls:
- "Add Item" button appends a new blank item row
- Drag handle on the left to re-order
- Delete button (X) on the right — if only 1 item left, show warning "You need at least 1 item"
- Validation: title and description both required per item before Submit is allowed

Type-specific rules displayed as hints below the item list:
- Top List: "Add up to 25 items. 10 is the sweet spot."
- This vs That: "Exactly 2 items required."
- Who Is Better: "2–5 candidates."
- etc.

### Step 4: Sources (Global)
Separate from per-item sources. For adding overall references:
- List of `{ label, url }` pairs
- "Add Source" button
- Drag to reorder
- Marking at least one valid URL: automatically toggles `is_sourced = true` on the post

> The composer shows: "✅ Sourced badge will be applied" when at least one URL is provided — this is motivational and encourages citation.

### Step 5: Preview + Submit
Full read-only preview of the entire post as it will appear on the post page. Then:
- A "Submit for Review" button for standard posts  
- For Counter Lists: "Publish Now" (no review, goes live immediately)

On submit:
- Disable all buttons, show loader
- Call `POST /api/posts`
- On success → redirect to `/post/{id}` with a success banner "Your post is in review 🎉"
- For Counter Lists → redirect immediately (live)
- On error → show error inline at Step 5 with the backend's `detail` message
